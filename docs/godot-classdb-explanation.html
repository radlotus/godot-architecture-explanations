Godot’s ClassDB: The Reflection System

This document explains Godot’s ClassDB, the engine’s powerful, low-level reflection system. ClassDB is not a system that game developers typically interact with directly, but it is the single most important piece of architecture that enables the entire Godot editor and the GDScript language to function.

Core Principles

  1. A Centralized Class Registry: ClassDB is a global singleton that acts as a database or “phone book” for every class registered with the engine. At startup, it knows about every class, what it inherits from, and what it can do.
  2. Reflection is Key: Reflection is the ability of a program to examine, introspect, and modify its own structure and behavior at runtime. ClassDB allows any part of the engine to ask questions like, “What properties does a Sprite2D have?” or “Does AnimationPlayer have a method called play?”
  3. Enabling the Editor: The Godot editor is not a separate application; it is a Godot game that makes heavy use of ClassDB. The Inspector, the scene tree, the code editor, and other tools are all built on top of the reflection data provided by ClassDB.
  4. Bridging C++ and Scripting: ClassDB is the bridge that allows a dynamic, interpreted language like GDScript to seamlessly interact with the engine’s core C++ classes.

The Architectural Building Blocks

The ClassDB Singleton

ClassDB (core/object/class_db.h) is a singleton that primarily contains a large HashMap where keys are the names of classes (StringName) and values are ClassInfo structs.

The ClassInfo Struct

This internal struct holds all the metadata for a single class:

  • Inheritance: A pointer to the ClassInfo of its parent class.
  • Properties: A list of all registered properties, including their name, type, and hint strings for the editor.
  • Methods: A map of all registered methods. This doesn’t just store the name, but a MethodBind object, which is a highly optimized C++ object for calling the method directly.
  • Signals: A map of all signals the class can emit.
  • Constants & Enums: A list of all exposed integer constants and enumerations.
  • Constructor: A function pointer to a C++ template function that can create a new instance of this class.

The Registration Process

Classes are not automatically known to ClassDB. They must be explicitly registered.

  1. The GDCLASS Macro: In the header file of every Godot class (e.g., class Sprite2D : public Node2D { GDCLASS(Sprite2D, Node2D); ... }), this macro injects the necessary boilerplate code for registration.

  2. The _bind_methods Function: Each class has a static C++ function called _bind_methods. This function is called only once when the engine starts up. Inside this function, the developer explicitly registers every property, method, and signal they want to expose to the reflection system.

    // Simplified example from a C++ source file
void MyNode::_bind_methods() {
  // Expose a method to be used in GDScript and the editor
  ClassDB::bind_method(D_METHOD("my_function", "arg1"), &MyNode::my_function);

  // Expose a property with a getter and setter
  ClassDB::add_property("my_property", PropertyInfo(Variant::INT, "my_property"), "set_my_property", "get_my_property");

  // Expose a signal
  ClassDB::add_signal("my_signal", MethodInfo("my_signal", PropertyInfo(Variant::STRING, "some_data")));
}

How is ClassDB Used?

  • The Inspector: When you select a node, the Inspector asks ClassDB for that class’s property list. It then iterates over the list and dynamically creates the appropriate UI controls (a text field for a String, a slider for an integer with a range hint, etc.). When you change a value in the Inspector, it calls the property’s registered setter method.
  • Scene Serialization: When saving a scene (.tscn), the engine queries ClassDB for the property list and their default values. It compares the node’s current values to the defaults and only saves the properties that have been changed.
  • GDScript: When you write my_node.my_property = 10, the GDScript interpreter asks ClassDB for the setter of "my_property" on the my_node’s class and calls it. This is how dynamic, string-based access works so efficiently.

Translating ClassDB to the Web

While JavaScript has some built-in reflection, it is not as structured as ClassDB. To build a tool or editor for the web, you would create your own, simplified version of this pattern. The goal is to make your components self-describing.

The Static Metadata Pattern

The cleanest way to achieve this is by adding static methods or properties to your JavaScript classes that define their own interface.

Example Self-Describing Component:

class PlayerComponent {
  constructor() {
    this.speed = 100;
    this.playerName = "Hero";
  }

  jump(height) {
    console.log(`Jumping ${height} units!`);
  }

  // --- Static Metadata for our ClassDB ---

  static getProperties() {
    return {
      speed: { type: 'number', default: 100, hint: 'range(0, 500)' },
      playerName: { type: 'string', default: 'Hero' }
    };
  }

  static getMethods() {
    return {
      jump: { args: ['height'] }
    };
  }

  static getSignals() {
    return {
      health_depleted: { args: [] }
    };
  }
}

The ClassDB Implementation

You would create a singleton manager to act as your ClassDB.

class ClassDB {
  constructor() {
    this.classes = new Map();
  }

  register(className, classDef) {
    this.classes.set(className, {
      constructor: classDef,
      properties: classDef.getProperties ? classDef.getProperties() : {},
      methods: classDef.getMethods ? classDef.getMethods() : {},
      signals: classDef.getSignals ? classDef.getSignals() : {},
    });
  }

  getClassInfo(className) {
    return this.classes.get(className);
  }

  instantiate(className) {
    const info = this.getClassInfo(className);
    if (info) {
      return new info.constructor();
    }
    return null;
  }
}

const classDB = new ClassDB();

// Register your components at startup
classDB.register("PlayerComponent", PlayerComponent);

// --- How an Inspector would use it ---

function createInspectorFor(object) {
  const className = object.constructor.name;
  const classInfo = classDB.getClassInfo(className);
  if (!classInfo) return;

  const inspectorDiv = document.createElement('div');
  
  for (const propName in classInfo.properties) {
    const propInfo = classInfo.properties[propName];
    // Create a label
    const label = document.createElement('label');
    label.textContent = propName;
    
    // Create an input based on the type
    const input = document.createElement('input');
    input.type = propInfo.type; // e.g., 'number', 'string'
    input.value = object[propName];

    // When the input changes, update the actual object property
    input.onchange = (e) => {
      object[propName] = e.target.value;
    };

    inspectorDiv.appendChild(label);
    inspectorDiv.appendChild(input);
  }

  document.body.appendChild(inspectorDiv);
}

const player = classDB.instantiate("PlayerComponent");
createInspectorFor(player);

By adopting this pattern, you create a powerful foundation for building development tools, editors, and serialization systems, just like Godot does.
