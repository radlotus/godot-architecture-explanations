Godot’s Signal System Architecture

This document explains Godot’s Signal system, the engine’s implementation of the Observer design pattern. Signals are the primary mechanism for creating decoupled, event-driven code, allowing different parts of your application to communicate without having direct references to each other.

Core Principles

  1. Decoupling: The object that emits a signal (the “emitter”) has no knowledge of the objects that receive it (the “listeners”). It simply broadcasts that an event has occurred. This allows you to create highly reusable components.
  2. Event-Driven: Signals allow you to react to events rather than constantly checking for state changes in the main loop (_process). This leads to cleaner, more efficient, and more organized code.
  3. Many-to-Many Communication: Any number of listeners can connect to a single signal, and a single object can listen to signals from many different emitters.
  4. Defined in Object: The signal mechanism is built into the base Object class, meaning any Godot object can emit or connect to signals, not just nodes.

The Architectural Building Blocks

The entire Signal system is defined within core/object/object.h.

The Internal Mechanism

Every object in Godot has a hidden map that stores its signal data.

  • signal_map: This is a HashMap where the keys are the names of the signals (e.g., "pressed", "body_entered") and the values are SignalData structs.
  • SignalData: This struct holds a list of all the functions that are listening to that specific signal.
  • Connection: Each entry in that list is a Connection object. A Connection simply stores the Callable (the listener’s object and method name) that needs to be executed when the signal is fired.

The Three Key Operations

  1. Defining a Signal: You declare a signal with the signal keyword in GDScript. In C++, this is done via ADD_SIGNAL. This creates an entry in the object’s signal_map.

  2. Connecting (Listening): When you call emitter.connect("signal_name", my_listener_callable), you are adding a new Connection object to the signal_name list inside the emitter’s signal_map.

  3. Emitting (Broadcasting): When you call emitter.emit_signal("signal_name", arg1, ...):

    1. The emitter looks up "signal_name" in its signal_map.
    2. It gets the list of all Connection objects for that signal.
    3. It iterates through the list and, for each Connection, it calls the stored Callable.
    4. The arguments you passed to emit_signal are forwarded to the listener’s method.

This simple, robust design allows for powerful and flexible communication between objects without creating hard dependencies.

Translating the Signal System to the Web

The observer pattern is ubiquitous in JavaScript. You can replicate Godot’s Signal system very easily using either native browser APIs or a simple, custom class.

The Native Browser Way: EventTarget

The browser has a built-in class for this exact purpose called EventTarget. All DOM elements (HTMLElement) already inherit from it.

  • connect -> addEventListener: This is the direct equivalent.
  • emit_signal -> dispatchEvent: You create a CustomEvent to send your signal.
// Any class can become a signal emitter by extending EventTarget
class MyButton extends EventTarget {
  click() {
    console.log("Button emitting 'pressed' signal.");
    
    // Create a custom event. The 'detail' property can hold arguments.
    const event = new CustomEvent("pressed", {
      detail: { pressTime: Date.now() }
    });

    // Dispatch the event to all listeners.
    this.dispatchEvent(event);
  }
}

const button = new MyButton();

// A listener object
const player = {
  onButtonPressed: (event) => {
    console.log(`Player received 'pressed' signal! Time: ${event.detail.pressTime}`);
  }
};

// Connect the signal
button.addEventListener("pressed", player.onButtonPressed);

// Emit the signal
button.click();

The Lightweight Custom Way: EventEmitter Class

For maximum clarity and to avoid relying on DOM-specific classes, you can write a simple EventEmitter class from scratch. This perfectly mimics Godot’s internal signal_map.

Example EventEmitter Implementation:

class EventEmitter {
  constructor() {
    // This is the equivalent of Godot's `signal_map`
    this.events = {};
  }

  /**
   * Subscribe to a signal.
   * @param {string} eventName - The name of the signal (e.g., "pressed").
   * @param {Function} callback - The function to call when the signal is emitted.
   */
  on(eventName, callback) {
    if (!this.events[eventName]) {
      this.events[eventName] = [];
    }
    this.events[eventName].push(callback);
  }

  /**
   * Unsubscribe from a signal.
   * @param {string} eventName - The name of the signal.
   * @param {Function} callback - The specific callback to remove.
   */
  off(eventName, callback) {
    if (!this.events[eventName]) return;

    this.events[eventName] = this.events[eventName].filter(
      (listener) => listener !== callback
    );
  }

  /**
   * Emit a signal, calling all subscribed listeners.
   * @param {string} eventName - The name of the signal to emit.
   * @param {...any} args - Arguments to pass to the listeners.
   */
  emit(eventName, ...args) {
    if (!this.events[eventName]) return;

    // Call each listener with the provided arguments.
    this.events[eventName].forEach(callback => {
      callback(...args);
    });
  }
}

// --- How to use it ---

// Your game objects can inherit from it or use it via composition.
class MyButton extends EventEmitter {
  press() {
    console.log("Button emitting 'pressed' signal.");
    this.emit("pressed", "some_data", 123);
  }
}

const button = new MyButton();

const player = {
  onPressed: (data, number) => {
    console.log(`Player received 'pressed' signal with: ${data}, ${number}`);
  }
};

// Connect (`on` is the equivalent of `connect`)
button.on("pressed", player.onPressed);

// Emit the signal
button.press();

This EventEmitter approach provides a clean, self-contained, and highly reusable way to implement the powerful decoupling pattern that Signals provide in Godot.
