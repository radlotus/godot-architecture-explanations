Godot-Inspired Web Engine: Design and Planning (DOM-Based)

This document outlines the architecture and project structure for a 2D game engine on the web, built on the principles of the Godot Engine. This plan specifically targets the DOM as the primary renderer, using HTML elements and CSS for visuals.

1. Core Philosophy

The engine will be built on the following core Godot philosophies:

  • Scene-Based: Everything is a scene, which is a tree of nodes. The scene tree maps directly to the DOM tree.
  • Node-Based Composition: Functionality is built by composing Node objects. Each Node will be a JavaScript class that manages a corresponding HTMLElement.
  • Decoupled Communication: Nodes will use signals (EventEmitter) to communicate without creating hard dependencies.
  • Centralized Managers (Servers): Global singleton managers will handle loading, input, and scene management, keeping the node logic clean.
  • Data-Driven: Scenes and resources will be defined in simple data files (JSON), separate from the game logic.

3. Core Class Breakdown (DOM-Based)

Node.js

  • Responsibility: To be the base class for all objects in the scene tree. It will now create and manage an HTMLElement.
  • API:
    • element: The HTMLElement this node manages (e.g., a <div>).
    • constructor(element_type = 'div'): Creates the DOM element.
    • addChild(node): Appends the child node’s element to its own element.
    • removeChild(node): Removes the child node’s element.
    • _process(delta): The logic update, to be overridden.
    • _render(): A new lifecycle method called after _process, responsible for applying state to the DOM element’s style.

Node2D.js (inherits from Node)

  • Responsibility: A node with 2D spatial properties.
  • API:
    • position = {x: 0, y: 0}
    • rotation = 0 (in radians)
    • scale = {x: 1, y: 1}
  • Interaction: Its _render() method will apply its position, rotation, and scale to its element.style.transform.

Sprite.js (inherits from Node2D)

  • Responsibility: To display an image.
  • Interaction: It will be constructed with an <img> element instead of a <div>. It will have a texture property that, when set, updates the element.src attribute.

Label.js (inherits from Node2D)

  • Responsibility: To display text.
  • Interaction: It will manage a <div> or <p> element. A text property will update the element.textContent.

The Render Step (No Renderer class)

We no longer need a dedicated Renderer class that draws to a canvas. The browser is our renderer. The “render step” is now the process of synchronizing our node state with the DOM. This will be handled by the new _render() lifecycle method on each node.

4. Data Flow Plan (DOM-Based)

A. Application Start-Up

  1. main.js is executed.
  2. The Engine object is created, instantiating the singleton managers.
  3. engine.start("path/to/main.scene.json") is called.
  4. SceneManager uses ResourceManager to fetch the scene JSON.
  5. SceneManager recursively builds the node tree. For each node definition in the JSON:
    1. It creates an instance of the specified JS class (e.g., new Sprite()).
    2. This in turn creates an HTMLElement (e.g., <img>).
    3. The element is appended to the parent’s element in the DOM.
  6. _enterTree() and _ready() are called on all nodes.
  7. The MainLoop is started.

B. A Single Game Frame

  1. MainLoop._update() is called by the browser.
  2. Time manager updates delta time.
  3. InputManager is updated.
  4. The fixed physics step is executed (if applicable, likely using a 3rd party library).
  5. SceneManager.update(delta) is called, which iterates through the scene tree and calls _process(delta) on every node.
    • A PlayerController script reads from InputManager and updates its parent Node2D’s position property.
    • The position property is just a variable in a JS object; no DOM interaction happens here.
  6. After the _process pass is complete, SceneManager.render() is called. This iterates through the tree and calls _render() on every node.
    • The Player node’s _render() method reads its final position, rotation, etc., and updates its element.style.transform property.
    • This batching of all DOM reads (_process) followed by all DOM writes (_render) is more efficient and helps avoid unnecessary browser layout recalculations.
  7. InputManager.endOfFrame() is called.
  8. The browser paints the updated DOM. The loop repeats.
